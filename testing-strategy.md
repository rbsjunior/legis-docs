# LEGIS Testing Strategy

## Overview

LEGIS is a domain-heavy application whose core value lives in the bill approval workflow — multi-role state transitions, approval paths, SME routing, proxy assignments, and audit logging. The testing strategy reflects that: integration tests over unit tests, real Postgres over mocks, and E2E focused on role-based critical paths.

**Stack:** Vitest (unit), Neon branch databases (integration), Playwright (E2E)

---

## Tooling Decisions

| Layer | Tool | Rationale |
|---|---|---|
| Unit runner | Vitest | Native ESM, no complex transform config for Next.js 15 |
| Integration DB | Neon branches | Real Postgres, no Docker, instantaneous branching |
| E2E | Playwright | Best Next.js support, parallel execution, reliable auth state handling |
| Auth mocking | Playwright stored state files | Login once per role, reuse across all E2E tests |
| CI DB isolation | Branch per CI run | Zero bleed between parallel runs |
| E2E target | Localhost + Neon branch | Isolated, fast, no circular dependency with staging |
| Staging validation | Scheduled smoke tests | Post-deploy only — not PR-gating |

---

## Value-to-Effort Ranking

### 1. Workflow Integration Tests — *Critical value, High effort, High complexity*

**What:** Test `actions/workflow.ts` and `actions/bills.ts` directly against a real Neon branch database.

**Why highest priority:** The bill lifecycle (DRAFT → submitted → routed → SME review → approved → enrolled) with approval paths, proxy assignments, and audit log entries is the entire product. Unit tests cannot cover this; only real Postgres exposes issues with Prisma queries, enum handling, and relational integrity.

**Covers:**
- Bill state transitions and guard conditions
- Approval path routing logic (Sr. Deputy assignment)
- SME assignment and review decisions
- Proxy assignment and proxied approval behavior
- Audit log entry creation on every workflow action
- Concurrent approval scenarios

**Infrastructure needed:** Neon branch per CI run, seed factory (see below).

---

### 2. E2E Critical Role Paths — *High value, High effort, High complexity*

**What:** Playwright tests against a locally-started Next.js server backed by a Neon branch. One critical path per role.

**Why:** Role-based UI differences (what a Staff member sees vs. a Sr. Deputy vs. an SME) can only be verified at the browser level. The 40+ components in `bills/[id]/_components/` and the multi-step approval flow require end-to-end coverage.

**Auth approach:** One stored state file per role, generated by a global Playwright setup that logs in once before any test runs.

| Role | State file | Critical path |
|---|---|---|
| Staff | `playwright/.auth/staff.json` | Create bill → submit for routing |
| Sr. Deputy | `playwright/.auth/sr-deputy.json` | Receive routed bill → approve/reject |
| SME | `playwright/.auth/sme.json` | Receive SME assignment → submit review |
| Admin | `playwright/.auth/admin.json` | Manage users, org, approval paths |
| Proxy | `playwright/.auth/proxy.json` | Act on behalf of assigned user |

**Important:** Stored state files are signed with `AUTH_SECRET`. Files generated against one environment will not authenticate against another environment with a different secret.

**E2E target:** `localhost:3000` started by Playwright's `webServer` config, connected to a Neon branch. Never target Railway staging for PR-gating (see Staging Smoke Tests below).

---

### 3. Admin Action Integration Tests — *Medium-high value, Medium effort, Medium complexity*

**What:** Test `actions/admin.ts` against a real Neon branch.

**Covers:**
- User creation with role assignment
- Org hierarchy (Administration → Bureau → Division → Section)
- Approval path configuration
- Role changes and their downstream effects

**Why before API routes:** Admin data (users, orgs, approval paths) is a prerequisite for all other tests. Getting this right enables the seed factory.

---

### 4. API Route Integration Tests — *Medium value, Medium effort, Medium complexity*

**What:** Test `/api/documents/upload`, `/api/documents/[id]`, `/api/pdf/bill/[id]`, `/api/pdf/routing/[id]`.

**Covers:**
- File upload validation and S3 write (mock S3 at the SDK boundary, not the route)
- Document retrieval and authorization
- PDF generation for bill and routing sheet

**Note:** Mock Resend and S3 at the client level — routes should be tested with real Next.js request/response handling, not with the external services.

---

### 5. Zod Schema & Validation Unit Tests — *Medium value, Low effort, Low complexity*

**What:** Unit tests for all Zod schemas used in Server Action input validation.

**Covers:**
- Required field enforcement
- Enum membership validation
- String length and format constraints
- Cross-field validation rules

**Why:** Fast to write, zero infrastructure, immediate feedback on bad input handling. Good starting point before CI is wired up.

---

### 6. Auth & Middleware Unit Tests — *Medium value, Low effort, Low complexity*

**What:** Unit tests for auth configuration and middleware redirect logic.

**Covers:**
- Login with valid/invalid credentials
- Role-based redirect rules in `middleware.ts`
- Session shape and expiry handling
- Admin route protection

---

### 7. Pure Utility Unit Tests — *Low value, Very low effort, Very low complexity*

**What:** Unit tests for pure functions in `lib/`.

**Note:** LEGIS has very little pure logic to unit test — most value lives in the DB-coupled layers above. Do not over-invest here.

---

### 8. Staging Smoke Tests — *Low value for PRs, Medium effort, Low complexity*

**What:** A small Playwright suite that runs post-deploy against the Railway staging URL.

**Covers:**
- Login succeeds
- Dashboard loads
- Database connection is alive
- At least one protected route is accessible

**Why not for PRs:** Targeting Railway staging for PR-gating creates a circular dependency — staging runs `main`'s code, not the PR's code, so tests pass regardless of what the PR contains. Concurrent PRs also share the same staging database, causing interference and flakiness.

**Schedule:** Trigger post-deploy via Railway webhook or a scheduled GitHub Actions run.

---

## The Seed Factory — Highest-Leverage Investment

All integration and E2E tests in rows 1–4 require valid test data. LEGIS's schema has deep relational requirements: a bill needs an org, users with specific roles and org assignments, and approval paths. Creating this ad-hoc in every test file is unmaintainable.

**Build one programmatic seed factory first.** It should produce:

- A complete org hierarchy (Administration → Bureau → Division → Section)
- One user per role, assigned to the correct org level
- A bill in any requested workflow state (DRAFT, SUBMITTED, ROUTED, SME_REVIEW, APPROVED, ENROLLED)
- The correct `BillApprovalPath` and `BillSme` records for that state
- The expected `AuditLog` entries for all transitions to reach that state

Every test then calls the factory to get a known starting state rather than building it from scratch.

## CI Architecture

```
PR opened
    │
    ├── Neon branch created (unique name: legis-ci-{run_id})
    │       │
    │       ├── prisma migrate deploy
    │       ├── seed factory (base data)
    │       │
    │       ├── Unit tests (Vitest) ──────────── parallel
    │       ├── Integration tests (Vitest) ─────── parallel
    │       └── E2E tests (Playwright)
    │               next build && next start
    │               against the Neon branch
    │
    └── Neon branch deleted (always, even on failure)

Post-deploy to Railway staging
    └── Smoke tests (Playwright against staging URL)
```

**CI requirements:**
- `NEON_API_KEY` — for branch creation/deletion
- `NEON_PROJECT_ID` — Neon project identifier
- `AUTH_SECRET` — must match the value used when generating stored auth state files
- Branch connection string injected as `DATABASE_URL` for the test run

**Branch cleanup:** The delete step must use `if: always()` so it runs even when tests fail.

## Key Risks

| Risk | Mitigation |
|---|---|
| Neon free tier branch cap (10) | Upgrade to paid plan before setting up CI |
| Auth state files go stale | Regenerate in CI global setup on every run; never commit real session tokens |
| Resend fires real emails during E2E | Mock `resend.emails.send` at the module boundary in test config |
| S3 writes during E2E | Mock `@aws-sdk/client-s3` in Playwright test environment |
| Parallel PR runs interfere | Each run uses its own Neon branch; E2E uses localhost, not staging |
| Seed factory becomes stale after migrations | Seed factory is a first-class artifact — update it alongside schema migrations |
